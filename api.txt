ZYRA API KEY REFERENCE

This file explains:
- which API keys / external credentials are used in this project
- what each one is used for
- where to get it
- which `.env` variable to update


1. MongoDB Atlas

Used for:
- main database
- applications collection
- users collection
- tests collection
- jobs collection

.env keys:
- MONGO_URI
- MONGO_DB

What you need:
- MongoDB Atlas connection string
- database name

Where to get it:
- MongoDB Atlas dashboard
- open your cluster
- click Connect
- choose Drivers
- copy the connection string

Official docs:
- https://www.mongodb.com/docs/guides/atlas/connection-string/
- https://www.mongodb.com/docs/current/reference/connection-string/

Example:
- MONGO_URI=mongodb+srv://<username>:<password>@cluster0.xxxx.mongodb.net/?retryWrites=true&w=majority
- MONGO_DB=zyra

Important:
- replace `<username>` and `<password>`
- make sure your IP/network is allowed in Atlas


2. Cloudinary

Used for:
- resume PDF upload
- storing uploaded files
- returning secure resume URLs

.env keys:
- CLOUDINARY_CLOUD_NAME
- CLOUDINARY_API_KEY
- CLOUDINARY_API_SECRET
- CLOUDINARY_FOLDER

Where to get it:
- log in to Cloudinary Console
- find your Cloud name in Dashboard / Product Environment
- open Settings -> API Keys
- copy Cloud name, API key, and API secret

Official docs:
- https://cloudinary.com/documentation/developer_onboarding_faq_find_credentials
- https://cloudinary.com/documentation/finding_your_credentials_tutorial
- https://support.cloudinary.com/hc/en-us/articles/202520942-Access-key-management-adding-and-removing-API-keys-and-secrets

Example:
- CLOUDINARY_CLOUD_NAME=your_cloud_name
- CLOUDINARY_API_KEY=123456789012345
- CLOUDINARY_API_SECRET=your_api_secret
- CLOUDINARY_FOLDER=zyra_resumes


3. Gmail SMTP

Used for:
- shortlist email with credentials
- rejection email
- virtual interview completion email

.env keys:
- SMTP_SERVER
- SMTP_PORT
- SMTP_USER
- SMTP_PASS

Recommended values:
- SMTP_SERVER=smtp.gmail.com
- SMTP_PORT=587

Where to get it:
- use your Gmail address for `SMTP_USER`
- for `SMTP_PASS`, use a Google App Password
- App Passwords require 2-Step Verification on your Google account

Official docs:
- https://support.google.com/mail/answer/185833?hl=en

Example:
- SMTP_SERVER=smtp.gmail.com
- SMTP_PORT=587
- SMTP_USER=yourgmail@gmail.com
- SMTP_PASS=your_16_digit_app_password

Important:
- do not use your normal Gmail login password here
- use an App Password instead


4. Hugging Face

Used for:
- MCQ generation
- virtual interview evaluation
- AI text generation fallback

.env keys:
- HF_TOKEN
- HF_API_URL
- MODEL
- VIRTUAL_HF_MODEL
- MCQ_SECONDARY_MODEL
- MCQ_TERTIARY_MODEL

Where to get it:
- log in to Hugging Face
- go to Settings -> Access Tokens
- create a new token
- for production, fine-grained tokens are preferred

Official docs:
- https://huggingface.co/docs/hub/security-tokens

Example:
- HF_TOKEN=hf_xxxxxxxxxxxxxxxxx
- HF_API_URL=https://api-inference.huggingface.co/v1/chat/completions
- MODEL=meta-llama/Llama-3.1-8B-Instruct
- VIRTUAL_HF_MODEL=meta-llama/Llama-3.1-8B-Instruct

Important:
- choose models that support your intended API usage
- some models may require access approval on Hugging Face


5. Ollama

Used for:
- optional local AI generation
- local MCQ / virtual question generation in development

.env keys:
- OLLAMA_URL
- OLLAMA_MODEL
- MCQ_OLLAMA_MODEL
- MCQ_USE_OLLAMA
- USE_LOCAL_VIRTUAL_MODEL

Where to get it:
- Ollama does not require a cloud API key by default
- you install Ollama locally and pull a model manually

Official site:
- https://ollama.com/

Example:
- OLLAMA_URL=http://localhost:11434/api/generate
- OLLAMA_MODEL=llama3.1
- MCQ_OLLAMA_MODEL=llama3.1

Important:
- this is mainly useful for local development
- for cloud deployment, API-based models are usually easier


6. D-ID

Used for:
- AI avatar question video generation

.env keys:
- DID_API_KEY
- DID_BASE_URL
- DID_AVATAR_SOURCE_URL
- DID_VOICE_PROVIDER
- DID_VOICE_ID
- DID_TALK_TIMEOUT_SECONDS

Where to get it:
- sign in to D-ID Studio
- open Account Settings
- generate API key

Official docs:
- https://docs.d-id.com/docs/api-keys

Important format note:
- D-ID shows the key in `API_USER:API_PASSWORD` format
- the app sends it in the `Authorization` header as Basic auth

Example:
- DID_API_KEY=your_api_user:your_api_password
- DID_BASE_URL=https://api.d-id.com
- DID_AVATAR_SOURCE_URL=https://create-images-results.d-id.com/DefaultPresenters/Noelle_f/image.png
- DID_VOICE_PROVIDER=microsoft
- DID_VOICE_ID=en-US-JennyNeural


7. Flask Secret

Used for:
- Flask session security

.env key:
- FLASK_SECRET

Where to get it:
- this is not from a third-party service
- generate your own strong random string

Example:
- FLASK_SECRET=change_this_to_a_long_random_secret_value

Important:
- use a long random value in production


8. Admin Credentials

Used for:
- recruiter/admin login

.env keys:
- ADMIN_USER
- ADMIN_PASS

Where to get it:
- you create these yourself
- they are not issued by an external provider

Example:
- ADMIN_USER=hr@zyra.com
- ADMIN_PASS=Admin@1234

10. Which Service Does What

MongoDB Atlas:
- stores application data and results

Cloudinary:
- stores uploaded resumes

Gmail SMTP:
- sends system emails

Hugging Face:
- AI question generation and evaluation

Ollama:
- optional local AI alternative

D-ID:
- avatar video generation


11. Safe Update Process

If you want to change any API key:
1. Create the new key from the provider dashboard
2. Update the matching `.env` value
3. Restart the app
4. Test the related feature
5. Revoke the old key if no longer needed


12. Quick Mapping

Resume upload not working:
- check Cloudinary keys

Emails not sending:
- check SMTP_USER and SMTP_PASS

Database not loading:
- check MONGO_URI and MONGO_DB

MCQ / virtual AI not generating:
- check HF_TOKEN, model names, or Ollama config

Avatar question video not working:
- check DID_API_KEY and DID_* settings


13. Production Advice

- never hardcode keys in source files
- keep them only in `.env` locally
- for cloud deployment, move them into a secret manager
- rotate leaked or shared keys
- use separate test and production credentials where possible
